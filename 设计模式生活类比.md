# 23 种设计模式 · 生活类比速查表

> 用日常生活的场景帮助理解每个设计模式的**核心机制**，不是简单贴标签。

## 创建型（5种）— 关注"对象怎么创建"

| 模式 | 生活类比 | 类比点（为什么贴切） |
|-----|---------|-------|
| **工厂方法** | 连锁蛋糕店订蛋糕 | 总部规定"接受蛋糕订单"这个流程，北京分店做京式糕点、苏州分店做苏式糕点；父类定接口，**子类决定创建哪个具体对象** |
| **抽象工厂** | 装修风格选择 | 选"中式风格"→配套中式门窗+中式家具+中式灯饰；换"欧式风格"→整套全换；**保证产品族风格统一** |
| **建造者** | 盖房子 | 包工头指挥固定流程（打地基→砌墙→封顶→装修），换施工队得不同风格；**分步构建复杂对象** |
| **原型** | 印章盖章 | 刻好一个印章，要盖多少份盖多少份，**不用每次重新刻**；复制现有对象而非新建 |
| **单例** | 公司 CEO | 全公司只有一个 CEO，所有人都找同一个；**全局唯一实例** |

## 结构型（7种）— 关注"对象怎么组合"

| 模式 | 生活类比 | 类比点（为什么贴切） |
|-----|---------|-------|
| **适配器** | 电源转换插头 | 国标插头插不上欧标插座，转换器在中间转换；**接口不兼容时转换** |
| **桥接** | 笔与墨水 | 笔的类型（钢笔/圆珠笔）和墨水颜色（黑/蓝/红）**两个维度独立变化**，任意组合 |
| **组合** | 公司组织架构 | 部门和员工都当作"组织单元"统一处理；**递归组合成树** |
| **装饰器** | 给咖啡加料 | 基础咖啡→加奶→加糖→加奶盖，每层包裹增强功能，**可任意组合顺序且透明** |
| **外观** | 政务大厅一站式窗口 | 不用分别跑工商、税务、银行，一个窗口办全套；**屏蔽子系统复杂度** |
| **享元** | 围棋下棋 | 棋盘 361 个位置，但只需"黑子""白子"两个共享对象，**位置（外部状态）由外部传入**，棋子本身（内部状态）共享 |
| **代理** | 房产中介 | 你不直接见房东，通过中介看房签合同；**代理控制对目标的访问** |

## 行为型（11种）— 关注"对象怎么通信"

| 模式 | 生活类比 | 类比点（为什么贴切） |
|-----|---------|-------|
| **责任链** | 请假审批 | 组长→主管→经理→老板，层层上报，**谁能处理谁处理** |
| **命令** | 餐厅点单 | 点单变成订单（命令对象），服务员提交、厨师执行；**请求封装成对象，可排队、可撤销** |
| **解释器** | 计算器解析表达式 | "3+5×2"按语法规则解析计算；**定义一种语法并解释执行** |
| **迭代器** | 图书馆书架找书 | 不管书横放竖放，统一"下一本"的方式遍历；**封装遍历逻辑，客户端不关心内部结构** |
| **中介者** | 塔台调度 | 飞机之间不直接通信，都通过塔台协调；**星形通信解耦，避免网状依赖** |
| **备忘录** | 游戏存档 | 存档保存当前状态，死了能读档恢复，**存档文件你改不了（不破坏封装）** |
| **观察者** | 订阅公众号 | 公众号发文，所有订阅者自动收到推送；**一对多通知** |
| **状态** | 自动售货机 | 投币前/投币后/出货中，同一按钮**在不同状态下行为不同** |
| **策略** | 导航选路线 | 同一目的地，可选最短/最快/避堵，**算法可随时切换** |
| **模板方法** | 做菜食谱 | 食谱规定步骤顺序（备料→热锅→下菜→调味），**咸淡口味由具体做菜的人定**；定义骨架，子类实现细节 |
| **访问者** | 医院看病 | 同一个病人，内科量血压、外科看伤口、化验科抽血；**元素类型固定，操作种类可扩展**（双重分派） |

## 三个最容易混淆的类比区分

| 易混淆 | 区别 |
|-------|------|
| **适配器 vs 外观** | 适配器是"一对一"转换接口；外观是"一对多"屏蔽子系统 |
| **代理 vs 装饰器** | 代理控制访问（中介不让见房东）；装饰器增强功能（咖啡加奶） |
| **状态 vs 策略** | 状态是"被动"切换（售货机状态自动流转）；策略是"主动"选择（用户选路线） |
| **工厂方法 vs 抽象工厂** | 工厂方法造"一个"产品；抽象工厂造"一族"配套产品 |
| **原型 vs 享元** | 原型是"复制一份新的"（盖印章）；享元是"共享同一个"（共用棋子对象） |

## 总结

每个模式都对应一个我们早就熟悉的现实场景——**设计模式不是发明新概念，而是把现实中解决问题的智慧提炼成代码结构**。

判断一个类比是否贴切的标准：**类比的"机制"是否和模式的"机制"一致**，而不是表面相似。
