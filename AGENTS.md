# AGENTS.md — 设计模式示例项目工作规范

> 本文件为 AI Agent 在本项目中工作时的指引，基于迭代器模式的实现经验提炼。

## 项目概述

- **目标**：用 C++17 实现 GoF 23 种设计模式的教学示例
- **分类**：创建型 / 结构型 / 行为型 三大类
- **每个模式**：一个独立目录，包含 `.cpp` 源码 + `README.md` 文档

## 目录结构

```
design_patterns/
├── creational/          # 创建型
│   └── builder/
├── structural/          # 结构型
│   ├── composite/
│   └── bridge/
├── behavioral/          # 行为型
│   ├── mediator/
│   └── iterator/
├── CMakeLists.txt       # 统一编译配置
├── README.md            # 项目总览（含模式清单与勾选状态）
└── AGENTS.md            # 本文件
```

## 每个模式的文件规范

```
[pattern_name]/
├── [pattern_name].cpp   # 示例源码（单一文件，自包含）
└── README.md            # 模式说明文档
```

### README.md 必须包含

1. **一句话总结** — 用一句话概括模式本质
2. **解决的问题** — 用代码对比展示"没有模式的问题" vs "有模式的解决"
3. **实现说明** — 类结构、关键设计决策
4. **编译运行命令**
5. **要点** — 核心价值点（3-4 条）
6. **与其他模式的关系**

### 代码规范

- **C++ 标准**：C++17（`-std=c++17`），不得使用 C++20 特性（如 `auto` 函数参数）
- **编译零警告**：`g++ -std=c++17 -Wall -Wextra` 无警告
- **现代 C++**：使用智能指针、移动语义、`constexpr` 等
- **自包含**：每个 `.cpp` 可独立编译运行，不依赖外部库
- **注释详尽**：文件头说明问题、思想、与其他模式对比

## 认知框架（基于迭代器模式梳理）

实现每个设计模式时，遵循以下认知框架：

### 1. 先讲清楚"解决什么问题"

用代码对比展示，不要空谈理论：

```
没有模式时：[具体场景代码] → [痛点]
有模式后：  [解决代码]   → [优势]
```

### 2. 表述要准确

- **避免绝对化**：不要说"所有代码都要改"，要说"客户端可能依赖存储特有操作，需要改"
- **区分必然与偶然**：存储变化必然要改迭代器实现，但客户端代码不变是模式带来的收益
- **用代码说话**：抽象结论必须配代码示例

### 3. 举例要严谨

- 代码示例必须逻辑自洽，经得起推敲
- 注释要说明具体表现，不能含糊（如"互相干扰"要说明具体干扰现象）

### 4. 多种实现风格对比

如果一个模式有多种实现风格（如迭代器的 GoF 风格 vs STL 风格），要对比说明：

- 各自的机制（运行时绑定 vs 编译期绑定）
- 本质相同点（都是把遍历逻辑委托给迭代器类）

## 编译验证

每完成一个模式，必须验证：

```bash
# 单文件编译
g++ -std=c++17 -Wall -Wextra -o [name] [name].cpp && ./[name]

# CMake 整体编译
cmake -B build && cmake --build build
```
